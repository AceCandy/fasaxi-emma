修复原则

排序按“先止血、再加护栏、最后收技术债”。

  1. 先处理会导致凭据泄露、未授权操作、错误日志扩散的问题，再谈结构优化。
  2. 代码修复和外部处置分开推进：凭据轮换、历史 jar 下架、日志留存排查不应等待代码合并。
  3. 每一阶段都必须有可验证验收条件，避免“改了但无法确认风险是否真的消失”。
  4. 涉及入口治理的改动先补测试再发版，避免把新鉴权和新校验做成二次事故。

阶段 1：48 小时内恢复发布安全边界

先把最危险的入口和凭据链路关住。这一阶段完成前，不建议继续对外发布。

1. 移除打包路径中的 secret

Critical · 1-2 天
  • 处理 application-local.yml、application-prod.yml、application-homac.yml 的打包路径，确保本地工作树配置不会进入 jar。
  • 清理 src/main/java/cn/acecandy/fasaxi/emma/utils/EmbyUtil.java 等硬编码凭据/常量。
  • 新增构建护栏：打包后自动检查 jar 内是否包含敏感配置或疑似密钥模式。
验收：jar tf target/fasaxi-emma.jar 不再出现本地敏感配置；仓库中不存在硬编码真实凭据；CI 对敏感内容可自动 fail。

2. 收紧管理接口暴露面

Critical · 1-2 天
  • 对 DebugController、ApiController 中的 /api/debug/**、/api/build/**、/api/clear/** 加管理员鉴权或仅限特定 profile。
  • 把带副作用的 GET 改为 POST/DELETE，避免被缓存、预取或误触发。
  • 同步复核 CorsConfig.java，避免跨域策略继续放大未授权调用面。
验收：非授权请求统一返回 401/403；副作用接口不再使用 GET；本地调试和生产暴露面有明确区分。

3. 日志脱敏止血

High · 2-4 小时
  • 修改 AccessLog.java、OriginReqService.java、EmbyContentCacheReqWrapper.java，统一掩码或白名单输出。
  • 禁止写入 api_key、Authorization、Cookie、完整请求头、完整查询串。
  • 为异常日志保留排障所需字段，但采用局部脱敏而不是整包落盘。
验收：压测和错误回放日志中不再出现敏感 header / token 明文。

阶段 2：一周内补齐运行护栏

目标是把“当前能跑”提升到“可以稳定发布”。

4. 修正 Redis 锁语义

High · 0.5-1 天
  • 重写 RedisLockClient.java：加 owner token、续期策略和 compare-and-delete 释放。
  • Redis 异常时显式失败或降级为“拒绝执行”，不要 silent fail-open。
  • 回归调用方：OriginReqService.java、VideoRedirectService.java。
验收：并发测试下无双执行、无误删他人锁；Redis 故障时行为可预测且有明确日志。

5. 补输入校验和错误边界

Medium · 4-8 小时
  • 对 debug/build 入口参数增加枚举或白名单校验，非法 cloud 参数直接返回 400。
  • 不要让非法输入流入 CloudUtil.java 深层再以空指针爆炸。
  • 顺手统一错误响应文案，便于调用方定位。
验收：非法参数路径有稳定 4xx 响应，不再出现深层 NPE。

6. 统一定时任务门禁与注释

Medium · 2-4 小时
  • 统一 Webdav2CloudTask.java、ShortDramaTask.java、TmdbProviderTask.java 的 enable gate。
  • 修正 cron 注释与真实周期不一致的问题，消除误导性运行文案。
  • 为禁用环境增加明确启动日志，避免“看起来没开，实际上在跑”。
验收：关闭总开关后任务不执行；注释、日志、真实周期三者一致。

7. 补关键入口测试并修正 Java 25 测试工具链

High · 1-2 天
  • 新增控制器/过滤器/配置打包链路测试，重点覆盖鉴权、日志脱敏、任务门禁、jar 产物卫生。
  • 保留现有 service 测试，同时补 DebugController、过滤器、打包断言的集成测试。
  • 处理 Mockito 在 Java 25 下的 agent 告警，避免未来升级直接失效。
验收：mvn test 覆盖关键安全入口；打包校验进入自动化；测试工具链告警有明确处置方案。

推荐并行方式

避免多人同时修改同一条入口链路。

  • A 线：凭据治理 + 打包护栏 + 历史产物排查。
  • B 线：Debug/Build/Clear 接口鉴权、HTTP 方法治理、参数校验。
  • C 线：日志脱敏 + Redis 锁修复 + 关键回归测试。
  • 定时任务门禁和文档修正可在 B/C 线完成后合并，避免反复改验收基线。

发布闸门

满足以下条件后再考虑恢复稳定发布。

  • 新凭据已完成轮换。
  • jar 产物不再携带本地敏感配置。
  • 管理接口具备鉴权或 profile 门禁。
  • 日志已完成脱敏。
  • 关键入口回归测试为绿。

首批执行清单

如果按最小 blast radius 开始做，建议按这个顺序落地。

  1. 先轮换 secret，并冻结当前对外发版。
  2. 再修 jar 打包和硬编码凭据问题，防止新构建继续扩散。
  3. 随后收紧 debug/build/clear 接口，并补参数校验。
  4. 接着做日志脱敏与 Redis 锁修复，同时补对应回归测试。
  5. 最后统一任务门禁、修正 README/banner、安排结构和性能债拆分。
