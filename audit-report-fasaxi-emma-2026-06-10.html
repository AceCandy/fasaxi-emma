📝 总体评估

📋 总体评估

项目概况：这是一个基于 Spring Boot 3.5.8 / Java 25 的单体代理与任务调度服务，职责横跨 Emby 反向代理、图片/视频重定向、Redis 缓存与分布式锁、多库访问、云盘编排以及多条后台同步任务。请求入口主要分为两条：`/api/**` 直接进入控制器，其他流量统一由 `EmbyProxyFilter` 包装并分发到 `OriginReqService`、`PicRedirectService`、`VideoRedirectService` 和 `VirtualService`。

主要风险：当前最严重的问题不是“代码不好看”，而是权限边界和密钥边界同时过宽。高权限调试/维护接口裸露在 HTTP 面上，本地凭据又能随工作树打进 `jar`，再叠加日志直接写出 `api_key` 与完整请求头，已经越过“发布前修一下”的范围，属于需要先收口暴露面再谈上线稳定性的状态。

亮点：入口分流、外部客户端和持久层大体仍保持了 `control/config/service/task/sao/dao` 分层；Redis、Feign、Hikari、Jetty 都配置了显式超时或容量上限；最近对 `VideoRedirectService`、`OriginReqService`、`PicRedirectService` 和 `EmbyConfig` 补了一批回归测试，`mvn test` 与 `mvn -DskipTests package` 当前都可以通过。

优先修复：第一优先级应是移除打包可见 secrets、给 `/api/debug/**` 和 `/api/build|clear/**` 加鉴权与 profile 门禁、统一改成非 GET 副作用接口、对日志做彻底脱敏；第二优先级是把 Redis 锁改成 owner-token + compare-and-delete，并补齐控制器/过滤器/配置层的集成测试。

总评：C 总评。仓库已经具备“本地能编、部分关键链路有回归测试”的基础，但距离“稳定公开发布”还有明显差距，主要卡在安全面、入口治理和发布卫生，而不是单个业务函数的细枝末节。

4. Redis 锁实现既 fail-open 又无 owner 校验，互斥语义不成立

High 已确认
📁 稳定性 / Fallback 📍 src/main/java/cn/acecandy/fasaxi/emma/sao/client/RedisLockClient.java:96-135；OriginReqService.java:101-111, 140-143；VideoRedirectService.java:101-114 🎯 RedisLockClient / OriginReqService / VideoRedirectService
问题：`RedisLockClient.lock()` 在 Redis 不可用、`setIfAbsent` 返回 `null`、甚至 key 为空时都直接返回 `true` 放行；`unlock()` 则只是简单 `delete(key)`，没有 owner token，也没有 compare-and-delete。调用方 `OriginReqService` 与 `VideoRedirectService` 都把它当成真正的互斥锁使用。这意味着 Redis 故障时会完全失去并发收敛，锁过期后旧请求的 finally 还能删掉新请求刚拿到的锁。
RedisLockClient.java:96-106: success == null || success catch (Exception) { return true; } RedisLockClient.java:130-134: redisTemplate.delete(key) OriginReqService.java:101-111 / VideoRedirectService.java:101-114 都在 finally 中无条件 unlock
最小修复
1. `lock()` 返回 owner token，不允许 Redis 异常静默放行。 2. `unlock()` 改成 Lua compare-and-delete。 3. 调用方对拿锁失败返回显式 429/503 或可观察的降级状态，不要伪装成正常执行。
⏱ 预计工作量： 0.5-1 天  ·  回归建议： 增加并发测试，覆盖 Redis 异常、锁过期后旧请求解锁、新旧请求交错三种场景，断言不会出现双执行或误删锁。

6. 线程限流决策在播放热路径上全量扫描 Redis key，并用 `synchronized` 串行化

Medium 已确认
📁 性能 📍 src/main/java/cn/acecandy/fasaxi/emma/utils/ThreadLimitUtil.java:45-120；RedisClient.java:329-338 🎯 ThreadLimitUtil.limitThreadCache / RedisClient.scanKeysByPrefix

⏱ 预计工作量： 0.5-1 天  ·  回归建议： 增加基准测试或并发测试，比较 10/100/1000 设备 key 下的 `limitThreadCache()` 耗时，确保不再随 key 数线性上升。

9. API 输入边界缺少校验，非法 `cloud` 参数会在 CloudUtil 深处触发空指针

📁 类型安全 / 后端 API 📍 pom.xml:84-87；DebugController.java:59-81；CloudStorageType.java:29-39；CloudUtil.java:128-146, 206-218, 230-255 🎯 DebugController / CloudStorageType / CloudUtil
